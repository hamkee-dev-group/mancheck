# mc_tests – Unified test engine for mancheck

This directory contains a self-contained test engine for the **mancheck** project.
It runs end-to-end checks against the analyzer binary and the specdb SQLite database,
and also integrates the existing `.c` / `.exp` golden tests under the top-level
`tests/` directory.

It is meant to live at the top level of the repository, alongside:

- `analyzer/`
- `specdb/`
- `tests/`
- `tools/`
- `vendor/`
- `mc_tests/`  ← this directory

## Layout

- `run.sh`  
  Main unified test runner. Builds the analyzer if needed, runs a suite of DB-based
  tests, specdb checks, a `--dump-views` sanity check, and then runs all golden
  `.c` / `.exp` tests from the top-level `tests/` directory.

- `tests/` (inside `mc_tests/`)  
  Small curated C source files used as inputs for the DB-based analyzer tests:

  - `test01_simple_unchecked.c` – unchecked `read`/`write`
  - `test02_no_unchecked.c` – properly checked `read`/`write`

- `out/` (created at runtime)  
  Temporary outputs produced by the tests:
  - `mancheck_test.db` – test SQLite database populated by DB-based analyzer runs
  - `views_test.jsonl` – JSONL file generated by `--dump-views`
  - `golden_test.db` – SQLite DB used when running golden tests

## Prerequisites

The runner assumes:

- You run it from the **repository root** (where `analyzer/`, `specdb/`, etc. live)
- `mc_tests/` is located directly under the repository root
- `sqlite3` is installed and available in `PATH`
- `make` is available
- The `specdb` database exists at `specdb/data/spec.db`
- Your existing golden tests live under the top-level `tests/` directory as
  `.c` + `.exp` pairs (e.g. `tests/foo.c` and `tests/foo.exp`)

On a typical dev box this just means:

```sh
cd /path/to/mancheck
ls
# analyzer/  specdb/  tests/  tools/  vendor/  mc_tests/ ...
```

## What the unified tests do

The suite currently covers:

### 1. Analyzer basic unchecked-return test (DB)

- Input: `mc_tests/tests/test01_simple_unchecked.c`
- Asserts via SQLite:
  - a `runs` row is created for that file
  - `error_count` for that run is `2`
  - exactly two `facts` are recorded for that run
  - one fact for `read`, one for `write`

### 2. Analyzer no-issues test (DB)

- Input: `mc_tests/tests/test02_no_unchecked.c` (returns checked)
- Asserts:
  - a `runs` row is created for this file
  - `error_count` for that run is `0`
  - no `facts` are recorded for that run

### 3. specdb core functions

- Queries `specdb/data/spec.db` using `sqlite3`
- Asserts that the following functions exist with reasonable sections:
  - `read(2)`
  - `write(2)`
  - `malloc(3|3p)`
  - `fopen(3|3p)`
  - `socket(2|3p|7)`
  - `pthread_create(3|3p)`

### 4. Analyzer `--dump-views`

- Runs the analyzer with `--dump-views` on `test01_simple_unchecked.c`
- Asserts that:
  - `views_test.jsonl` is created
  - the file is non-empty
  - the first line looks like a JSON object

### 5. Golden `.c` / `.exp` tests

- For each `tests/*.c` in the **top-level** `tests/` directory:
  - expects a companion `tests/*.exp` file
  - runs the analyzer from the repository root with:

    ```sh
    ./analyzer/analyzer --db mc_tests/out/golden_test.db tests/<name>.c
    ```

  - captures combined stdout/stderr
  - compares it against the corresponding `.exp` file with `diff -u`

- On match: logs `golden: output matches for <file>` and counts as a PASS  
- On mismatch: logs `golden: MISMATCH for <file>`, prints the unified diff,
  and counts as a FAIL

This replicates your original `.c` / `.exp` harness while using the current
analyzer path and `--db` flag, so the expectations stay in sync with the
new binary.

If there is no `tests/` directory, or no `tests/*.c` files, the golden
section is simply skipped.

## How to run

From the **top-level** of the `mancheck` repo:

```sh
cd /path/to/mancheck
./mc_tests/run.sh
```

You should see something like:

```text
[INFO]  Project root: /home/crypto/devel/mancheck
[INFO]  === Test 1: analyzer basic unchecked-return (DB) ===
...
[INFO]  === Test 5: golden .c/.exp tests (text output) ===
    ==> foo.c
[PASS]  golden: output matches for foo.c
...
All tests passed (passed=..., failed=0)
```

The script exits with status `0` on success and `1` on failure, so it is
safe to use in CI.

## Makefile integration

You can hook this into the top-level `Makefile` with a `test` target:

```make
.PHONY: all analyzer test clean

all: analyzer

analyzer:
    $(MAKE) -C analyzer

test: analyzer
    ./mc_tests/run.sh

clean:
    $(MAKE) -C analyzer clean || true
    rm -f mc_tests/out/mancheck_test.db mc_tests/out/views_test.jsonl mc_tests/out/golden_test.db
```

Adjust paths as needed for your existing build system.

## Adding more tests

### DB-based tests

- Drop additional `.c` files under `mc_tests/tests/`.
- Add new `test_...` functions in `run.sh` that:
  - run the analyzer on those files
  - query `mc_tests/out/mancheck_test.db` with `sqlite3`
  - call `expect_eq` with meaningful conditions
- Register the new functions in `main()`.

### Golden `.c` / `.exp` tests

- Add more `.c` / `.exp` pairs under the **top-level** `tests/` directory.

  Example:

  - `tests/my_case.c`
  - `tests/my_case.exp`

- The unified runner will automatically pick them up as long as the names
  match (`my_case.c` ↔ `my_case.exp`).

This way you get both styles:

- DB-level tests that assert on the internal facts/runs tables.
- Text-golden tests that assert on the precise diagnostic output the user sees.
